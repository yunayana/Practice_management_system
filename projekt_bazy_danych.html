Zadanie 2. Szkielet tabel w SQL

Na podstawie diagramu ERD przygotowano plik baza_danych.sql zawierający instrukcje CREATE TABLE dla wszystkich 10 tabel. Poniżej przedstawiono kluczowe fragmenty schematu z omówieniem decyzji projektowych.

2.1 Tabela users – baza użytkowników

CREATE TABLE users (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    full_name     TEXT    NOT NULL,
    email         TEXT    NOT NULL UNIQUE,
    password_hash TEXT    NOT NULL,
    role          TEXT    NOT NULL CHECK (role IN
                    ('student','uczelniany','zakladowy','sekretariat')),
    is_active     INTEGER NOT NULL DEFAULT 1,
    created_at    TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at    TEXT    NOT NULL DEFAULT (datetime('now'))
);

2.2 Tabela practices – główna encja dokumentu

CREATE TABLE practices (
    id                       INTEGER PRIMARY KEY AUTOINCREMENT,
    student_id               INTEGER NOT NULL,
    company_name             TEXT    NOT NULL,
    start_date               TEXT    NOT NULL,
    end_date                 TEXT    NOT NULL,
    hours                    INTEGER,
    status                   TEXT    NOT NULL DEFAULT 'Draft'
                             CHECK (status IN ('Draft','Submitted',
                             'Under_Review','Rejected','Approved','Closed')),
    university_supervisor_id INTEGER,
    company_supervisor_id    INTEGER,
    grade                    TEXT,
    created_at               TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at               TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY (student_id)               REFERENCES students(id),
    FOREIGN KEY (university_supervisor_id) REFERENCES supervisors(id),
    FOREIGN KEY (company_supervisor_id)    REFERENCES supervisors(id)
);

2.3 Encja pośrednia dla relacji N:M

CREATE TABLE effects_confirmation_items (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    confirmation_id INTEGER NOT NULL,
    effect_id       INTEGER NOT NULL,
    confirmed       INTEGER NOT NULL DEFAULT 0 CHECK (confirmed IN (0,1)),
    UNIQUE (confirmation_id, effect_id),
    FOREIGN KEY (confirmation_id) REFERENCES effects_confirmations(id),
    FOREIGN KEY (effect_id)       REFERENCES learning_effects(id)
);
Pełny schemat wszystkich tabel (CREATE TABLE + dane słownikowe dla learning_effects) znajduje się w pliku baza_danych.sql.

2.4 Zasady projektowania schematu

  • Klucze główne: wszystkie tabele używają INTEGER PRIMARY KEY AUTOINCREMENT
  • Klucze obce: zdefiniowane jako FOREIGN KEY...REFERENCES we wszystkich tabelach zależnych
  • Integralność: pola obowiązkowe oznaczone NOT NULL, unikalność przez UNIQUE
  • Ograniczenia domenowe: pola role i status walidowane przez CHECK
  • Kolumny techniczne: created_at i updated_at z wartością domyślną datetime('now') w każdej tabeli
  • Relacja N:M: zrealizowana przez tabelę pośrednią effects_confirmation_items z kluczem złożonym UNIQUE(confirmation_id, effect_id)

Zadanie 3. Dobór systemu bazy danych

3.1 SQLite – wybór dla etapu prototypu

Na początkowym etapie projektu rekomenduje się SQLite jako silnik bazy danych. Uzasadnienie wyboru:

  • Jest wbudowana w standardową bibliotekę Pythona (moduł sqlite3 – brak dodatkowej instalacji)
  • Nie wymaga osobnego serwera bazodanowego – baza to pojedynczy plik .db
  • Bardzo prosta konfiguracja i uruchomienie w projekcie Flask
  • Wystarczająca wydajność dla prototypu i testów z kilkudziesięcioma użytkownikami
  • Możliwość łatwego wersjonowania bazy danych razem z kodem (plik .db w repozytorium)
  • Umożliwia szybkie opracowanie modelu danych oraz przetestowanie logiki biznesowej

3.2 Ograniczenia SQLite w kontekście dalszego rozwoju

SQLite ma istotne ograniczenia, które stają się problematyczne przy rozbudowie systemu:
  • Brak pełnej obsługi współbieżności – jednoczesne zapisy mogą powodować blokady (lock)
  • Ograniczone wsparcie dla ALTER TABLE (np. brak usuwania kolumn w starszych wersjach)
  • Brak uwierzytelniania użytkowników bazodanowych i zarządzania uprawnieniami
  • Nie nadaje się do wdrożenia wieloserwerowego ani na produkcji z wieloma równoczesnymi użytkownikami
  • Ograniczona obsługa typów danych (brak natywnego BOOLEAN, DATE, UUID)

3.3 Docelowa alternatywa: MariaDB

Dla wersji produkcyjnej lub przy skalowaniu systemu rekomenduje się przejście na MariaDB (fork MySQL). Uzasadnienie:

  • Pełna obsługa współbieżności i transakcji ACID
  • Zarządzanie użytkownikami i uprawnieniami na poziomie bazy danych
  • Lepsza wydajność przy większej liczbie użytkowników i dużych zbiorach danych
  • Szerokie wsparcie hostingów i serwisów chmurowych (AWS RDS, Azure, GCP)
  • Łatwa migracja z SQLite – schemat SQL jest w dużej mierze kompatybilny (drobne różnice składni)

Zadanie 4. Narzędzie do prototypowania bazy danych: DBeaver Community

4.1 Opis narzędzia

DBeaver Community to darmowe, wieloplatformowe narzędzie do zarządzania bazami danych, dostępne na Windows, macOS i Linux (dbeaver.io). Jest rekomendowane jako alternatywa dla MySQL Workbench ze względu na szersze możliwości i lepszą kompatybilność z różnymi silnikami baz danych.

4.2 Kluczowe funkcje w kontekście projektu

  • Obsługa wielu silników bazodanowych – SQLite, MySQL, MariaDB, PostgreSQL, Oracle i inne w jednym narzędziu
  • Edytor SQL z podświetlaniem składni, autouzupełnianiem i historią zapytań
  • Przeglądanie schematu bazy danych – graficzna prezentacja tabel, kolumn, indeksów i kluczy obcych
  • Tworzenie i edycja diagramów ER na podstawie istniejącego schematu (funkcja „ER Diagram")
  • Import i eksport danych w formatach CSV, SQL, JSON, Excel
  • Podgląd i edycja danych tabel bezpośrednio w interfejsie graficznym
  • Wsparcie dla migracji – można połączyć się z SQLite i MariaDB w jednym oknie i porównać schematy

4.3 Zastosowanie w projekcie

Etap projektuZastosowanie DBeaver
PrototypowanieTworzenie i testowanie schematu SQLite, weryfikacja kluczy obcych
Weryfikacja danychPodgląd zawartości tabel podczas testów aplikacji Flask
Generowanie ERDAutomatyczne generowanie diagramu ER z istniejącego schematu SQL
MigracjaŁączenie się z MariaDB i przenoszenie danych z SQLite
DokumentacjaEksport schematu do SQL i diagramów dla celów dokumentacyjnych

4.4 Jak połączyć DBeaver z bazą projektu

  • Pobrać DBeaver Community ze strony dbeaver.io (bezpłatny)
  • Nowe połączenie → wybrać SQLite jako sterownik bazy danych
  • Wskazać plik .db projektu w katalogu aplikacji Flask
  • W zakładce „Database Navigator" przeglądać schemat tabel i dane
  • Prawy klik na bazę → View Diagram → automatyczny diagram ER z istniejącego schematu

Zadanie 5. Narzędzie do organizacji projektu bazy danych: Mermaid

5.1 Opis narzędzia

Mermaid to open-source'owe narzędzie do tworzenia diagramów w składni tekstowej (podobnej do Markdown). Diagramy są generowane automatycznie z kodu tekstowego, co eliminuje potrzebę ręcznego rysowania i ułatwia wersjonowanie wraz z kodem źródłowym projektu.

5.2 Uzasadnienie wyboru

  • Integracja z GitHub – GitHub natywnie renderuje bloki Mermaid w plikach README.md (widoczne już w tym projekcie)
  • Wersjonowanie – diagram jako kod tekstowy można przechowywać w repozytorium Git i śledzić zmiany
  • Wiele typów diagramów – ERD, sekwencja, stany, flowchart, Gantt (wszystkie potrzebne w projekcie)
  • Brak potrzeby instalacji – dostępny online (mermaid.live), w VS Code i w GitHub
  • Łatwość modyfikacji – zmiana diagramu to edycja tekstu, nie ręczne przesuwanie elementów
